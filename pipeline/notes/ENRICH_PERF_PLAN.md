# План: производительность enrich.ts

## Сначала факты (замер на этой машине, Node, те же данные)

- rabbits/ = 73 JSON-файла, 72 МБ. Полный load + JSON.parse: **~1.7 с**, heap
  ~150 МБ.
- Построение lowercase-индекса всех 27 entries-файлов: **~100 мс**.
- Lookup по прогретому кэшу — hashmap-доступ, микросекунды.

Вывод: архитектура enrich.ts (ленивая загрузка + кэши `entriesCache` /
`synsetCache` / `lowerCache`) уже правильная. 8000 строк ≈ полный прогрев всех
кэшей + 8000 дешёвых lookup'ов → **ожидаемое время единицы секунд**, не минуты.
Прежде чем что-то менять — сделать замер на Deno (`console.time` вокруг init и
основного цикла, прогон ALL.clean.csv).

## Улучшения по убыванию отдачи

### 1. Параллельная предзагрузка (холодный старт)

Сейчас файлы читаются лениво и последовательно — по мере встречи
букв/lexname'ов. Заменить на явный warm-up в `init()`: `Promise.all` на все
entries-* (и synset-файлы, если известно, что вход покрывает всё). I/O
распараллелится, парсинг останется ~1.7 с, но исчезнут «ступеньки» в середине
прогона. Выигрыш небольшой, код простой.

### 2. Прекомпилированный hypernym-индекс (главный резерв для --hypernyms)

`buildGlobalIndex()` грузит все 45 synset-файлов (~31 МБ) только чтобы резолвить
`id → members[0]`. Сделать одноразовый скрипт (или флаг `--build-index`),
который выгружает компактный `rabbits/hypernym-lemmas.json`
(`{synsetId: lemma}`, ~2–3 МБ). Тогда `--hypernyms` перестаёт стоить полной
загрузки synsets: −1 с старта и −~100 МБ памяти.

### 3. Синхронный горячий путь

`enrich()` делает `await synsetFromSense()` на каждый sense — после прогрева это
cache-hit, но каждый await = обход microtask queue: ~50–100 тыс. лишних тиков на
8000 слов. После warm-up из п.1 кэши полны — сделать lookup'ы синхронными
(`Map.get` без промисов). Выигрыш десятки–сотни мс; делать только если замер
покажет, что это заметно.

### 4. Отчёт о прогрессе

Уже есть (`processed % 100`). Добавить время в лог (`rows/sec`), чтобы регрессии
были видны сразу.

## Что НЕ делать (не окупится на 8000 строк)

- **Workers / шардинг по буквам** — задача не CPU-bound, JSON уже в памяти;
  накладные на сериализацию съедят выигрыш.
- **Стриминг вывода** — 8000 строк результата спокойно живут в памяти.
- **Сортировка входа по первой букве** — кэш без вытеснения, локальность ничего
  не даёт.
- **SQLite / LMDB / бинарный индекс** — оправдано только если данные вырастут на
  порядок (500k+ строк входа или rabbits > 500 МБ). Тогда: SQLite с таблицами
  entries/synsets и индексом по lower(headword).
